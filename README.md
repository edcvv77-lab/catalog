# منصة مجتمع أيهم (Aiham Community Command Center)

هذه النسخة (V4) تحوّل مشروع الدردشة السابق إلى منصة مجتمع شاملة يمكنها العمل بالكامل على GitHub Pages مع Supabase. ستجد هنا نظامًا متكاملاً يجمع بين الدردشة، المنشورات، إدارة المشاريع والمهام، مركز المعرفة، التذاكر والدعم، الإنجازات والسمعة، البحث، الإشعارات، ولوحة مالك قوية لإدارة كل شيء. تم الحفاظ على جميع ميزات الدردشة السابقة وتحسينها، مع إضافة طبقات جديدة لإدارة المحتوى والمجتمع.

## ما الجديد في الإصدار الرابع (V4)

إلى جانب الدردشة المتقدمة ولوحة المالك، يضيف هذا الإصدار:

1. **نظام الملفات الشخصية المتقدم:** صفحات لعرض الملف الشخصي وتعديله، مع دعم الصورة الرمزية، النبذة، الحالة، تاريخ الانضمام، آخر ظهور، عدد الرسائل والمنشورات، الشارات، الرتبة، والروابط الشخصية.
2. **موجز المنشورات (Feed):** نشر منشورات نصية أو مع صورة، عرض المنشورات في موجز زمني، التعليق والتفاعل، تصنيف المنشورات، وتثبيتها من قبل المالك أو المشرفين.
3. **نظام المشاريع والمهام:** إنشاء مشاريع متعددة، إضافة مهام بأولويات وحالات مختلفة، تعيين مستخدمين، وعرض لوحة كانبان بسيطة.
4. **مركز المعرفة:** إضافة وتصنيف المقالات، دعم المقالات العامة أو الخاصة، وتفاعل المستخدمين معها.
5. **نظام التذاكر والدعم:** يسمح للمستخدمين بفتح تذاكر دعم لأنواع مختلفة من المشاكل، وإدارة التذكرة (تعيين، تغيير الحالة، الردود) من قبل المشرفين.
6. **نظام الإنجازات والسمعة:** حساب نقاط السمعة وعدد الرسائل والمنشورات، مع إمكانية عرض لوحة الصدارة للمستخدمين الأكثر نشاطًا.
7. **مركز الإشعارات:** إشعارات مدمجة للرسائل الخاصة والردود والتعليقات والتذاكر، مع صفحة مخصصة لإدارة الإشعارات وقرائتها.
8. **بحث موحد:** صفحة بحث تسترجع المستخدمين والغرف والمنشورات والمقالات في نتيجة واحدة.
9. **لوحة مالك موسّعة:** أقسام جديدة لإدارة المنشورات والتعليقات والمقالات والمشاريع والمهام والتذاكر والإعلانات والأدوار والصلاحيات وإعدادات المنصة والثيمات.

## الإصلاحات الرئيسية في هذا الإصدار

* **توحيد أسماء الأعمدة بين الكود وقاعدة البيانات:** تم تحديث الجداول في الملف `database/schema_v4_fixed.sql` حتى تتوافق مع ما يستخدمه الكود. على سبيل المثال:
  - جدول `knowledge_articles` يستخدم العمود `content` بدلاً من `body`.
  - جدول `support_tickets` يستخدم الأعمدة `title` و`type` بدلاً من `subject` و`category`.
  - تمت إضافة عمود `role` إلى جدول `announcements` لاستخدامه عند استهداف دور معيّن بالإعلان.
* **إضافة سياسات RLS متكاملة:** تم تفعيل RLS على جميع الجداول المهمة، وإعادة كتابة السياسات باستخدام عبارة `drop policy` ثم `create policy` بدلاً من `create policy if not exists` غير المدعومة في PostgreSQL/Supabase. تشمل هذه السياسات جداول الملفات الشخصية، الشارات، نقاط السمعة، الغرف، الرسائل العامة والخاصة، المنشورات، التعليقات، المهام، المقالات، التذاكر، الإعلانات، الإشعارات، الصلاحيات، وغيرها. تتم حماية كل جدول بحيث يمكن للمستخدم الوصول إلى سجلاته الخاصة فقط، ويمنح أصحاب الأدوار العليا (owner، admin، moderator) صلاحيات أوسع. يمكنك مراجعة جميع السياسات وتعديلها في نهاية ملف `database/schema_v4_fixed.sql`.

* **إضافة مُشغّل لإنشاء الملف الشخصي تلقائيًا:** تم إنشاء دالة `handle_new_user` ومُشغّل `on_auth_user_created` بحيث يتم إنشاء صف في جدول `profiles` تلقائيًا عند تسجيل مستخدم جديد في جدول `auth.users`. هذا يضمن عدم تعطل المنصة عند إنشاء حسابات جديدة دون الحاجة إلى إنشاء الملف الشخصي يدوياً في الواجهة.
* **تصحيح تحميل الملفات:** تم إزالة استخدام `type="module"` مع `utils.js` لأنه يتسبب في خطأ (Unexpected token export) عند تحميل الصفحات عبر GitHub Pages. الآن يتم تحميله كسكربت عادي مع ربط الدوال عبر `window.utils`. تمت إضافة تحميل `supabase-api.js` في `login.html` قبل `auth.js` لضمان توفر الطبقة البرمجية عند إنشاء الحسابات وتسجيل الدخول.
* **تصحيح روابط CSS وJS:** تم تصحيح المسارات الخاطئة مثل رابط التنسيق في `dm.html` وتوحيد استخدام مكتبة Supabase.
* **إضافة صورة أفتراضية للملفات الشخصية:** تم إضافة `assets/default-avatar.png` لمنع ظهور روابط صور مكسورة عندما لا يحدد المستخدم صورة.

## الإصلاحات النهائية (Secure Patch)

بعد إصدار النسخة **advanced_chat_platform_v4_final_secure** ظهرت بعض الثغرات البسيطة في صلاحيات إدراج الملفات الشخصية وفي إعدادات المالك. تم معالجة هذه الثغرات كما يلي:

1. **تقييد إنشاء الملف الشخصي (profiles_insert):** تم تحديث سياسة الإدراج لجدول `profiles` بحيث يَشترط أن يكون `id = auth.uid()`، وأن يكون الدور (`role`) هو `'user'` والحالة (`status`) هي `'active'` بشكل صريح. لا يمكن لأي مستخدم تحديد دور أعلى (مثل owner أو admin أو moderator) عند إنشاء ملفه الشخصي، ولا يمكن تغيير الحالة الافتراضية. هذا يمنع رفع الصلاحية عند التسجيل أو الإنشاء اليدوي. يبقى إنشاء الملف الشخصي يعمل تلقائياً عبر المُشغّل `handle_new_user`، ويعمل أيضًا عبر وظيفة upsert في الواجهة طالما تستوفي الشروط المذكورة.

2. **سياسات إدراج وتعديل إعدادات المالك (owner_settings):** تمت إضافة سياسة إدراج (`owner_settings_insert`) تسمح فقط لـ owner أو admin بإنشاء إعداد جديد. القراءة العامة للإعدادات تعتمد على قيمة `is_public = true`، بينما التعديل مسموح فقط لـ owner أو admin. لا يمكن للمستخدم العادي إنشاء أو تعديل أي إعداد. تم إضافة عمود `is_public` ليحدد ما إذا كان الإعداد ظاهراً للمستخدمين العاديين.

تم تحديث ملف `database/schema_v4_fixed.sql` ليعكس هذه التعديلات النهائية. عند تشغيل هذا الملف في Supabase، ستتم حماية الجداول بشكل صحيح ضد أي محاولة لرفع الصلاحيات أو التلاعب بالإعدادات.

## خطوات الإعداد (من البداية)

1. **إنشاء مشروع Supabase:** قم بتسجيل الدخول إلى [Supabase](https://supabase.com) وأنشئ مشروعًا جديدًا. سجّل قيم **Project URL** و **Anon public key** من إعدادات المشروع.
2. **ضبط ملف Supabase:** افتح `assets/supabase-config.js` وضع قيم `SUPABASE_URL` و`SUPABASE_ANON_KEY` الخاصة بمشروعك. **لا تشارك هذه القيم مع الآخرين.**
3. **تهيئة قاعدة البيانات:** في لوحة Supabase، افتح قسم **SQL Editor** ثم انسخ محتويات `database/schema_v4_fixed.sql` بالكامل والصقها وشغّلها. سيقوم ذلك بإنشاء جميع الجداول وإضافة المؤشرات وتفعيل سياسات RLS الأساسية. يمكنك مراجعة السياسات وتعديلها بحسب احتياجاتك.
4. **تفعيل Realtime:** انتقل إلى **Database → Replication** في لوحة Supabase وفعّل Realtime للجداول التي تحتاج تحديثًا لحظيًا مثل `messages`, `private_messages`, `posts`, `tasks`, `support_tickets` وغيرها.
5. **رفع الملفات إلى GitHub Pages أو Vercel:** ضع مجلد `advanced_chat_platform` (بما في ذلك `public` و`assets` و`database`) في مستودع GitHub أو أي منصة نشر. عند استخدام GitHub Pages يجب الانتباه إلى أن ملفات HTML داخل `public` تستورد الملفات من `../assets/`، وبالتالي لا يمكن تعيين `public` كجذر للموقع مباشرة؛ لأن المسار `../assets` لن يكون موجودًا وسيؤدي ذلك إلى كسر الروابط. أمامك حلّان:
   - إما نقل مجلد `assets` بالكامل داخل `public` وتعديل جميع المسارات في الصفحات (من `../assets/...` إلى `assets/...`)،
   - أو استخدام خدمة نشر مثل Vercel أو Netlify تسمح بتعيين مجلد المشروع بالكامل كجذر بحيث تبقى بنية المجلدات كما هي، ويتم تقديم `public/index.html` كنقطة دخول مع بقاء `assets` في المستوى الأعلى.
   من الأسهل عادةً نشر المشروع على Vercel، حيث يمكنك تحديد المجلد الجذر (`.`) وملف الدخول (`public/index.html`) وستتمكن المنصة من خدمة الملفات بشكل صحيح. إذا اخترت GitHub Pages، ستحتاج إلى إعادة هيكلة المسارات كما ذكر أعلاه حتى تعمل الروابط بشكل صحيح.
6. **تعريف أدوار المستخدمين:** بعد تشغيل المخطط، سجّل الدخول بحسابك الجديد. ثم من **SQL Editor** يمكنك جعل نفسك مالكًا بإجراء استعلام مثل:
   ```sql
   update profiles set role = 'owner' where id = '<YOUR_USER_ID>';
   ```
   يمكن الحصول على معرف المستخدم من جدول `auth.users`.
7. **اختبار كل صفحة:**
   - ابدأ بـ `login.html` لإنشاء حساب وتسجيل الدخول.
   - استخدم `chat.html` للدردشة وإنشاء غرف وتطبيق التثبيت والرد والتفاعلات.
   - جرّب `feed.html` لنشر منشور والتعليق عليه.
   - افتح `projects.html` ثم `tasks.html` لإنشاء مشروع ومهام.
   - جرّب `knowledge.html` لإضافة مقالة والتصنيف.
   - افتح `support.html` لإنشاء تذكرة دعم ثم تابعها عبر `ticket.html`.
   - استخدم `profile.html` و`profile-edit.html` لعرض وتعديل ملفك.
   - جرّب `leaderboard.html` لرؤية لوحة الصدارة، و`notifications.html` لقراءة الإشعارات، و`search.html` للبحث.
   - كمالك، استخدم `owner.html` للتحكم بكل شيء: المستخدمين، الغرف، المنشورات، المشاريع، التذاكر، الإعدادات، الثيم.

## الترقية من الإصدار الثالث إلى الإصدار الرابع

إذا كان لديك نسخة سابقة (V3 أو V3-Pro)، فإن الترقية تتطلب الآتي:

1. **أخذ نسخة احتياطية من بياناتك القديمة.**
2. **استبدال ملفات الواجهة:** قم بنسخ محتويات مجلد `public` و`assets` من هذا الإصدار إلى مشروعك، وتأكد من الاحتفاظ بملف `supabase-config.js` الخاص بك.
3. **تحديث قاعدة البيانات:** شغّل ملف `database/schema_v4_fixed.sql`. إذا كانت لديك بيانات موجودة في جداول قديمة، ستحتاج إلى ترحيلها يدويًا إلى البنية الجديدة (مثل نقل أعمدة `body` إلى `content`، أو `subject` إلى `title`).
4. **تعديل السياسات:** إذا كنت قد أنشأت سياسات RLS خاصة في الإصدار السابق، فراجعها وحدثها بحيث تتوافق مع الجداول الجديدة.

## الأمان والسياسات

* تم تفعيل **Row Level Security** على جميع الجداول المهمة، وتم إضافة سياسات أساسية (يمكنك العثور عليها في نهاية `schema_v4_fixed.sql`). هذه السياسات تسمح للمستخدمين بقراءة سجلاتهم الخاصة والتعديل عليها، وتمنح صلاحيات إضافية للأدوار (`owner`, `admin`, `moderator`). يجب مراجعة هذه السياسات وتخصيصها حسب احتياجاتك لضمان حماية البيانات.
* جميع مدخلات المستخدم تُعقم باستخدام `utils.sanitize()` لتفادي هجمات XSS. لا يتم إدراج أي محتوى إلى DOM باستخدام `innerHTML` إلا بعد التعقيم.
* يجب عدم مشاركة مفاتيح Supabase الخاصة أو مفتاح الخدمة (service role) في ملفات الواجهة. فقط `Anon Key` تُستخدم في الواجهة.
* لوحة المالك مخفية عن المستخدم العادي وتتحقق من الدور قبل تحميلها. أي محاولة للوصول مباشرة يجب أن ترفق بتحقق في الواجهة وسياسات RLS في الخلفية.

## حدود GitHub Pages + Supabase

هذه المنصة تعمل بشكل كامل باستخدام HTML وJavaScript في الواجهة وSupabase كخلفية. ومع ذلك هناك بعض القيود:

* **التخزين والملفات:** لا يوجد تحميل للصور أو الملفات مباشرة عبر GitHub Pages؛ إذا أردت رفع ملفات ستحتاج إلى استخدام Supabase Storage أو خدمة أخرى.
* **الأداء:** جلسات Supabase تعتمد على الاتصال بالإنترنت وقد تواجه بعض التأخير مقارنة بتطبيقات مقيّمة جانب الخادم.
* **التخصيص المتقدم للسياسات:** لا يمكن القيام بالإجراءات الإدارية المتقدمة مثل إنشاء إجراءات مخزنة مباشرة من الواجهة؛ يجب استخدام لوحة Supabase أو واجهة برمجية خارجية.

## ملف SQL النهائي

يحتوي `database/schema_v4_fixed.sql` على جميع الجداول المحدثة والسياسات والمؤشرات. هذا الملف هو الإصدار النهائي للإعداد الأمني، وقد تمت إزالة استخدام عبارة `create policy if not exists` واستُبدلت بعمليات `drop policy` ثم `create policy` لضمان التوافق مع Postgres/Supabase. كما يتضمن مُشغّل `handle_new_user` لإنشاء الملفات الشخصية تلقائياً. يجب استخدام هذا الملف بدلاً من أي ملف SQL سابق عند إعداد القاعدة. شغّله مرة واحدة فقط على قاعدة بيانات فارغة أو بعد ترحيل البيانات القديمة.
